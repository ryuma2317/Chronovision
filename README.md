## рҹ“– Overview

Chronovision is a full-stack platform that helps students and teachers act on
academic data. Students get a **predicted GPA** from a trained machine-learning
model, a **what-if simulator** to see how habits change that prediction, and an
**automatically generated study plan** built around their real free time.
Teachers manage classes, lessons, quizzes (including **AI-generated** questions
from uploaded material), attendance, and an **at-risk dashboard**. Admins manage
users and classes.

The system is split into three cooperating services: a **Python ML pipeline**
served by a **Flask microservice**, a **Node/Express + MySQL** REST API, and a
**React (Vite)** frontend.

---

## вңЁ Features

### For students
- **GPA prediction** from an XGBoost model trained on academic + lifestyle factors
- **What-if simulator** вҖ” adjust study hours, attendance, sleep, etc. and see the projected change
- **Two-stage study planner** вҖ” suggests weekly hours per subject toward a target GPA, then schedules them into your free-time slots (and refuses impossible plans)
- **Quizzes** with instant, server-graded results
- **Aptitude (IQ) test** that feeds the study planner
- **Lessons**, **attendance view**, and **gamification** (points, badges, leaderboard)

### For teachers
- **Class, lesson, and quiz management**
- **AI quiz generation** вҖ” upload a `.docx`/`.pdf` lesson and generate multiple-choice questions automatically
- **Attendance recording**
- **At-risk dashboard** and per-student detail with predictions

### For admins
- **User management** (create/edit/deactivate across roles)
- **Class management** (create classes, assign teachers, enroll students)

### Machine learning
- Synthetic dataset generator for safe, reproducible training data
- Five models compared (Linear Regression, Decision Tree, Random Forest, SVR, XGBoost)
- **XGBoost** selected as the production model, served via Flask

## рҹӨ– Machine learning

- **Dataset** вҖ” generated by `data_generation/generate_data.py`: synthetic
  student records combining academic history and lifestyle factors (study hours,
  attendance, sleep, diet, environment, aptitude, and more). The generated CSV is
  large and **git-ignored**; regenerate it any time.
- **Feature set** вҖ” a mix of numeric and categorical features (e.g. `gender`,
  `major`, `diet_quality`, `study_environment`), encoded at training time; the
  fitted encoders and feature list ship alongside the model
  (`encoders.pkl`, `feature_columns.pkl`).
- **Models compared** вҖ” Linear Regression, Decision Tree, Random Forest, SVR, and
  XGBoost (`Prediction_Model_Testing/`).
- **Production model** вҖ” XGBoost (`model/xgboost.pkl`), served by the Flask API.

The Flask service exposes:
- `POST /predict` вҖ” a flat feature object вҶ’ predicted GPA + risk status
- `POST /whatif` вҖ” `{ baseline, changes }` вҶ’ baseline vs. simulated GPA and the delta
- `GET /health`, `GET /features`

## рҹҡҖ Getting started

### Prerequisites
- **Node.js** 18+ and npm
- **Python** 3.10+
- **MySQL** 8+

### 1. Clone
```bash
git clone https://github.com/<you>/chronovision.git
cd chronovision
```

### 2. Database
```bash
mysql -u root -p < backend/chronovision_schema.sql
mysql -u root -p < backend/chronovision_seed.sql   # idempotent, safe to re-run
```

### 3. Backend API
```bash
cd backend
cp .env.example .env          # fill in DB credentials + JWT secrets (see below)
npm install
node scripts/create-admin.js "Your" "Name" you@school.edu "a-strong-password"
npm run dev                   # http://localhost:3001  В·  docs at /api/docs
```

### 4. Flask ML service (separate terminal)
```bash
cd Flask_API
pip install -r requirements.txt
python app.py                 # http://localhost:5001
```

### 5. Frontend (separate terminal)
```bash
cd frontend
npm install
npm run dev                   # http://localhost:5173
```

The frontend dev server proxies `/api` to the backend at `localhost:3001`, so no
frontend configuration is needed for local development.

### 6. (Optional) Regenerate data / retrain
```bash
cd data_generation && python generate_data.py
cd ../Prediction_Model_Testing && python model_xgboost.py
```

## рҹ”җ Environment variables

Set these in `backend/.env` (copy from `backend/.env.example`):

| Variable | Description | Default |
|---|---|---|
| `PORT` | Backend port | `3001` |
| `NODE_ENV` | Environment | `development` |
| `DB_HOST` / `DB_PORT` | MySQL host / port | `localhost` / `3306` |
| `DB_NAME` | Database name | `chronovision` |
| `DB_USER` / `DB_PASSWORD` | MySQL credentials | вҖ” |
| `JWT_SECRET` | Access-token signing secret | **required** |
| `JWT_EXPIRES_IN` | Access-token lifetime | `15m` |
| `JWT_REFRESH_SECRET` | Refresh-token signing secret | **required** |
| `JWT_REFRESH_EXPIRES_IN` | Refresh-token lifetime | `7d` |
| `FLASK_ML_URL` | Flask ML service base URL | `http://localhost:5001` |
| `ANTHROPIC_API_KEY` | Enables AI quiz generation (optional) | вҖ” |
| `UPLOAD_DIR` | Upload directory | `./uploads` |

The Flask service optionally reads `MODEL_DIR` to locate `xgboost.pkl`; by default
it finds the model in `Prediction_Model_Testing/model/`.

> **Never commit your real `.env`.** Use long, random JWT secrets.
> AI quiz generation is optional вҖ” leave `ANTHROPIC_API_KEY` blank and manual quiz
> creation still works.

## рҹ“Ў API overview

The backend is documented with Swagger at **`http://localhost:3001/api/docs`**.
Endpoints are grouped by role:

| Group | Examples |
|---|---|
| **Auth** | `POST /api/auth/login`, `POST /api/auth/refresh`, `GET /api/auth/me` |
| **Admin** | user management, class management |
| **Teacher** | classes, lessons, quizzes, attendance, at-risk dashboard |
| **Student** | lessons, quizzes, predictions, what-if, study plan, IQ test |

Authentication uses JWT access + refresh tokens (`Authorization: Bearer <token>`).

## вҡ пёҸ Known limitations

- The **aptitude (IQ) test** is a lightweight 15-question scoring quiz used to
  inform the study planner вҖ” **not** a validated psychometric instrument.
- **AI quiz generation** requires an Anthropic API key (usage-billed); it's
  optional and the app works fully without it.
- `multer@1.x` is used for uploads and is fine for a course project, but should be
  upgraded to `2.x` before any production use.
- Refresh tokens are not revoked server-side; a leaked token stays valid until it
  expires.
